# 管理台前端架构设计

## 背景

当前管理台前端集中在 `server/web/src/App.vue`、`server/web/src/style.css` 和少量业务组件中。页面能完成登录、图片管理、计划管理和预览操作，但职责边界不清晰，后续继续加功能会越来越难维护。

主要问题集中在以下几处：

- `App.vue` 同时负责应用壳、登录、图片上传、图片列表、备注编辑、计划列表、计划编辑、预览缓存和状态提示。
- `style.css` 同时承载全局基础样式、主题颜色、布局、按钮、表单、业务卡片和响应式规则。
- 当前只有 `PlanEditor.vue` 一个业务组件，图片管理、计划管理和通用控件没有形成组件体系。
- API 类型、请求封装、页面状态和对象 URL 生命周期混在页面逻辑中，页面代码难以局部调整。
- 界面视觉以卡片堆叠为主，缺少稳定的信息层级，图片管理和计划管理的主次关系不够清楚。

本次重构目标是整理管理台前端结构，让后续功能可以按模块推进，并沿用现有后端接口语义。

## 设计目标

管理台定位为设备内容维护工具，界面应该紧凑、清晰、便于反复操作。整体设计按以下目标收敛：

- 应用入口只负责组合页面壳、登录状态和主要工作区。
- 主题变量集中定义，颜色、间距、圆角、字号、边框、阴影和状态色统一管理。
- 基础组件只处理通用交互和视觉，不包含图片、计划等业务判断。
- 图片管理和计划管理拆成独立业务模块，模块内部再拆列表、表单、选择器和预览。
- API 请求、业务状态、对象 URL 缓存和提示消息分别收口，页面组件通过组合式函数使用。
- 桌面端优先保证表格化、列表化和扫描效率；移动端保证主要操作可以纵向完成。

## 目标目录

重构后的 `server/web/src` 建议按应用层、接口层、基础组件、业务模块、状态逻辑和样式层组织：

## 页面结构

管理台采用单页工作台结构，不引入路由。页面从上到下分为三块：

- 顶部栏：显示系统名称、管理员身份、刷新入口和退出入口。
- 概览栏：显示计划数量、图片数量、可显示图片数量、处理失败数量等关键数字。
- 工作区：使用图片管理和计划管理两个主区域，两个区域在同一页面内纵向排列。

图片管理承担资源维护：

- 搜索备注。
- 上传原始图片并填写备注。
- 查看图片处理状态。
- 修改备注。
- 查看 display 图片预览。

计划管理承担播放内容编排：

- 设置查询天数，默认三天，输入宽容收敛到一到七天。
- 新增、编辑、删除计划。
- 从已有图片中选择计划图片。
- 计划列表展示处理后的图片预览，处理未完成时显示对应状态。

图片上传入口归属图片管理，计划管理专注从已有图片中编排内容。

样式文件职责固定如下：

- `tokens.css`：只放设计变量。
- `base.css`：只放 reset、body、基础字体、原生元素默认样式。
- `layout.css`：只放应用壳、页面栅格、区域间距和响应式布局。
- `components.css`：只放基础组件共享样式。
- `utilities.css`：只放少量工具类，例如文本省略、哈希换行、屏幕阅读器文本。

业务组件优先使用组件内 scoped 样式。跨多个业务组件复用的视觉再上提到基础组件或 `components.css`。

## 基础组件

基础组件只描述通用 UI 行为，不感知图片、计划、token 或后端接口。

建议第一批建立以下组件：

- `BaseButton`：支持 `primary`、`secondary`、`ghost`、`danger`、`small`、`block`、`loading`。
- `BaseInput`：统一 label、错误提示、禁用状态、输入框高度。
- `BaseNumberInput`：用于计划天数，负责数值输入的显示一致性。
- `BaseFileInput`：封装文件选择样式和已选文件名称。
- `BasePanel`：承载区域标题、说明、右侧操作和内容区。
- `BaseToolbar`：统一搜索、筛选和操作按钮排列。
- `BaseBadge`：统一管理员、图片状态、数量标记。
- `BaseEmpty`：统一空状态。
- `BaseDialog`：用于后续确认删除、图片预览等弹层场景。

按钮、输入框、徽标和面板使用固定尺寸变量，减少页面因内容变化产生跳动。

## 业务组件

图片模块：

- `ImageManager`：图片管理模块入口，组合搜索、上传、列表和刷新。
- `ImageUploadPanel`：只负责上传表单，提交后返回文件和备注。
- `ImageList`：只负责列表结构和空状态。
- `ImageListItem`：展示单张图片的预览、哈希、状态、备注和操作。
- `ImagePreview`：根据 `sha256` 和预览 URL 展示图片或处理状态。
- `ImageRemarkEditor`：负责备注草稿、保存按钮和保存中状态。

计划模块：

- `PlanManager`：计划管理模块入口，组合天数筛选、编辑器和列表。
- `PlanEditor`：保留现有职责，但拆掉图片选择列表和重复工具函数。
- `PlanImagePicker`：只负责从已有图片中选择 `sha256`。
- `PlanList`：只负责计划列表结构。
- `PlanListItem`：展示计划标题、日期范围、图片数量、图片行和操作按钮。
- `PlanImageRow`：展示计划中的单张图片状态、备注、哈希和缩略图。

业务组件之间通过 props 和 emits 通信，本地存储由 `useAuth` 管理，请求拼接由 API 层管理。

## 状态与请求

API 层拆成统一请求客户端和资源文件：

- `api/client.ts`：统一处理 `baseUrl`、请求头、JSON envelope、错误消息。
- `api/types.ts`：集中定义 `ImageStatus`、`AdminImage`、`AdminPlan`、`PlanPayload`、`LoginResponse`。
- `api/auth.ts`：登录接口。
- `api/images.ts`：图片列表、上传、备注更新、display 图片读取。
- `api/plans.ts`：计划列表、新增、更新、删除。

组合式函数负责状态和副作用：

- `useAuth`：管理 `jwtToken`、`expiresAt`、登录、退出和本地存储。
- `useImages`：管理图片列表、关键词、上传、备注保存、刷新。
- `usePlans`：管理计划列表、天数、创建、更新、删除。
- `useImagePreviews`：管理 display 图片对象 URL、释放 URL、失败记录和批量预加载。
- `useToast`：统一成功和失败提示。
- `useAsyncState`：抽取 loading、error 和执行包装，减少重复 try/catch。

对象 URL 的创建和释放只放在 `useImagePreviews`，页面卸载和图片集合变化时统一回收。

## 交互细节

图片状态按照后端 `status` 字段显示：

- `pending`：待处理。
- `processing`：处理中。
- `ready`：可显示。
- `failed`：处理失败。

计划列表中的图片展示遵循同一状态口径。`ready` 显示 display 预览，其他状态显示状态文本。用户接口和管理员接口的数据差异由后端返回决定，前端管理台只使用管理员登录后的数据。

提示消息统一由 `AppToast` 展示。删除计划使用确认弹层，刷新、保存和上传失败统一显示后端返回的 `message`。

哈希展示统一使用短哈希加完整 title，列表中保持可换行，详情或后续复制场景再显示完整值。

## 响应式布局

桌面端以 1180px 内容宽度为上限，页面使用单列工作台，列表项内部使用稳定栅格：

- 图片列表项：预览、主体信息、操作区三列。
- 计划图片行：缩略图、主体信息、状态三列。
- 表单区域：日期和输入项使用两列，窄屏改为一列。

移动端保持同一页面结构，工具栏、上传表单、列表项操作区纵向排列。图片预览尺寸使用固定宽度和宽高比，保证加载前后布局稳定。

## 重构批次

后续实现建议分四批推进，每批都能单独验证：

- 主题与基础样式：建立 `styles` 目录、主题变量和基础样式入口，保持现有页面功能。
- 基础组件：抽出按钮、输入框、面板、工具栏、徽标和空状态，替换现有通用 HTML 结构。
- 状态与 API：拆分 `api` 文件和 `composables`，把登录、图片、计划和预览缓存从 `App.vue` 移出。
- 业务模块：拆出图片管理和计划管理组件，把 `App.vue` 收敛为应用入口和工作台组合。

每批完成后执行 `bun run build`，检查类型和前端构建。进入界面实现后，再启动 Vite 本地服务并用浏览器检查桌面和移动宽度下的主要页面。

## 验收标准

重构完成后，管理台应满足以下标准：

- `App.vue` 只承担应用入口和页面组合，图片和计划的业务细节归入对应模块。
- 全局 CSS 拆分为主题、基础、布局、组件和工具类，样式职责按文件归口。
- 图片管理和计划管理分别有清晰入口组件，计划管理中只选择已有图片。
- 登录、图片、计划、预览缓存和提示消息都有独立组合式函数。
- 图片状态在图片列表、计划编辑和计划列表中显示口径一致。
- 桌面端列表可扫描，移动端操作可纵向完成，按钮和文本不互相挤压。
- `bun run build` 通过。
