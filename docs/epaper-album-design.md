# Epaper Album 设计文档

## 项目目标

Epaper Album 是基于 Waveshare ESP32-S3-PhotoPainter 的个人电子墨水相册。设备从云端同步未来三天的照片计划，把图片缓存在 TF 卡中，并按日期或本地轮播规则刷新屏幕。

正常展示内容固定为一张照片、左下角标题和右下角日期。天气、日历、纪念日、海报等复杂内容由云端或人工提前制作成完整图片，设备仍按照片显示处理。

设备端固件使用 Rust 开发。代码以设备业务逻辑、网络同步、缓存管理、图片渲染、低功耗休眠和异常保护为核心，不引入 C/C++ 模块，也不通过 FFI 调用 C/C++ 驱动。Waveshare 工程作为硬件资料和行为参考，用于确认引脚、初始化流程、颜色编码、屏幕刷新流程、图片格式支持和功耗策略。

## 硬件基础

目标硬件为 ESP32-S3-PhotoPainter，主控为 ESP32-S3，屏幕为 7.3 英寸六色电子墨水屏，分辨率 800 x 480。硬件具备 Wi-Fi、BLE、PSRAM、TF 卡、RTC、温湿度传感器、音频相关外设和锂电池供电能力。

当前实测整屏刷新一张照片约 8 秒，参考工程也以整屏缓冲和整屏刷新为主。设备显示策略按低频、静态、必要刷新设计，同步、下载、状态确认等过程保留当前画面，通过日志和状态记录表达运行情况。

ESP32-S3 支持 deep sleep 定时唤醒。固件进入深度睡眠前配置 RTC timer，下一次唤醒时间由计划变化时间和同步时间共同决定。

## 技术路线

固件运行在 ESP-IDF 生态上，使用 Rust 实现设备逻辑和硬件控制。联网、HTTP、文件系统、时间、日志、休眠等能力优先使用 Rust 生态中的 ESP-IDF 绑定和服务库；屏幕驱动、图片解码、电源状态、TF 卡访问等能力按 Rust 模块实现。

设备端模块划分如下：

- `config`：读取和校验 `/sdcard/config.toml`。
- `schedule`：解析云端计划，选择当前显示项，计算下一次唤醒时间。
- `sync`：连接 Wi-Fi，请求计划，按 `sha256` 资源键下载缺失显示图片。
- `cache`：维护本地计划、资源索引、显示状态和空间保护清理。
- `render`：解码图片，合成标题、日期和左上角通知 sprite。
- `display`：管理六色电子墨水屏缓冲区、颜色编码和整屏刷新。
- `power`：读取电量状态，处理低电量策略，配置 deep sleep。
- `state`：保存运行状态，输出串口日志。

## 本地配置

设备配置文件放在 TF 卡根目录，路径固定为 `/sdcard/config.toml`。配置文件提供 Wi-Fi、云端接口地址和鉴权密钥。

```toml
wifi_ssid = "Home WiFi"
wifi_password = "password"
base_url = "https://example.com/epaper"
secret-key = "local-secret-key"
```

字段说明：

- `wifi_ssid`：Wi-Fi 名称。
- `wifi_password`：Wi-Fi 密码。
- `base_url`：云端接口基础地址。
- `secret-key`：请求云端接口时使用的鉴权密钥。

配置文件变更后，设备重启生效。配置缺失、字段缺失或解析失败时，设备显示整屏配置异常页。

## 云端接口

设备每天至少请求一次云端计划接口。请求携带配置文件中的 `secret-key`，服务端返回当前及未来几天的照片计划。设备端直接使用服务端接口结构，云端响应模型和设备同步模型保持一致。

计划接口：

```http
GET {base_url}/api/plans?days=3
secret-key: local-secret-key
```

成功响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": [
    {
      "id": 1,
      "start": "2026-06-06",
      "end": "2026-06-06",
      "caption": "晚风和海",
      "images": [
        "7f83b1657ff1fc53b92dc18148a1d65dfc2d4b1fa3d677284addd200126d9069"
      ]
    }
  ]
}
```

字段说明：

- `code`：接口状态码，`0` 表示成功。
- `message`：接口状态说明。
- `data[]`：设备可读取的计划列表。
- `data[].id`：计划 ID。
- `data[].start`：计划开始日期。
- `data[].end`：计划结束日期，包含当天。
- `data[].caption`：左下角标题。
- `data[].images`：已经处理完成的显示图片 `sha256` 列表。

设备按服务端返回结果覆盖本地计划文件。显示刷新判断使用当前日期、计划 ID、图片 `sha256`、图片序号和标题共同确定。

图片下载接口：

```http
GET {base_url}/images/{sha256}
secret-key: local-secret-key
```

图片下载成功时返回 BMP 二进制内容。设备使用计划中的 `sha256` 作为服务端显示图片资源键和本地缓存文件名，服务端通过 `images.status = 'ready'` 和 `data/images/display/{sha256}` 保证下载内容可用于显示。

文字 sprite 接口：

```http
GET {base_url}/api/sprite?type=caption&text=晚风和海
secret-key: local-secret-key
```

`type` 取值为 `caption`、`date`、`notice` 或 `status`。服务端根据 `server/assets/fonts.toml` 和字体目录生成白底黑字 BMP 小图块，字体、字号、fallback 顺序、内边距和字形栅格化规则由服务端统一管理。设备当前使用 `caption`、`date` 和 `notice`，分别用于左下角标题、右下角日期和左上角通知合成；`status` 作为右上角扩展类型保留。

设备端和服务端按同一组 sprite 术语处理：`caption` 是左下角标题，`date` 是右下角日期，`notice` 是左上角通知，`status` 是右上角扩展。服务端图片处理状态使用 `images.status` 字段表达。

## 显示资源

云端优先提供已经适配 800 x 480 的图片资源。设备侧以显示为主，减少裁剪、缩放和复杂图像处理。

资源格式按 PhotoPainter 官方资料收敛。参考工程支持 JPG、PNG、BMP，JPG 仅支持 Baseline DCT。云端优先输出 800 x 480 BMP，JPG 和 PNG 作为兼容输入格式。

设备所有正常展示图片都从 TF 卡缓存读取。缓存是正常显示路径，不触发异常通知。

文字 sprite 作为显示资源的一部分缓存到 TF 卡。计划同步成功后，设备为当前及未来计划标题预取 `caption` sprite，为显示日期预取 `date` sprite，并为常见通知预取 `notice` sprite。离线运行时优先使用本地已缓存 sprite 合成页面。

## 渲染规则

设备只渲染一种正常页面：照片页面。页面由图片、左下角标题、右下角日期和可选左上角通知组成。右下角日期由设备本地 RTC 当前日期生成，格式固定为 `YYYY-MM-DD`。右上角区域预留给后续 `status` 扩展，当前页面不合成右上角内容。

渲染顺序如下：

- 解码缓存图片到 800 x 480 屏幕缓冲区。
- 图片尺寸不匹配时按居中适配处理，空白区域使用白色。
- 标题非空时，加载或下载 `caption` sprite，并合成到左下角。
- 加载或下载设备当前日期对应的 `date` sprite，并合成到右下角。
- 存在通知时，加载或下载 `notice` sprite，并合成到左上角。
- 将最终缓冲区提交给屏幕做整屏刷新。

左下角标题、右下角日期和左上角通知的字体样式由服务端 sprite 生成配置统一控制。设备端负责读取 sprite BMP、按固定位置合成到 800 x 480 屏幕缓冲区，并保持边距、对齐和覆盖规则稳定。

sprite BMP 使用白底黑字。设备合成时把白色像素作为文字块背景，把黑色像素作为文字前景。标题为空时跳过标题 sprite 合成。sprite 缺失且当前无法下载时，设备保留照片主体显示，并在运行状态中记录缺失原因；整屏异常页仍只用于配置异常、TF 卡不可用和无可用照片。

## 启动与唤醒流程

设备进入应用后统一执行一次调度流程。冷启动、复位、烧录后启动、RTC 唤醒和手动按键唤醒使用同一条业务路径，唤醒原因进入串口日志和诊断信息。

- 初始化必要外设。
- 读取 `/sdcard/config.toml`。
- 读取本地计划、资源索引和显示状态。
- 判断当前时间、同步时间和计划生效状态。
- 到达同步时间时连接 Wi-Fi 请求计划。
- 保存服务端返回的计划列表。
- 按 `sha256` 资源键下载缺失显示图片。
- 下载或更新本轮需要的标题、日期和通知 sprite。
- 根据当前日期选择显示计划。
- 对比当前显示项和上次显示项。
- 内容变化时渲染并刷新屏幕。
- 内容未变化时保留当前屏幕。
- 计算下一次唤醒时间。
- 配置 deep sleep timer 并进入深度睡眠。

联网、下载和状态确认过程不刷新屏幕。只有最终显示内容变化、异常保护需要显示，或者通知随正常刷新写入时，设备才刷新电子墨水屏。

## 唤醒与同步

设备端固定每天至少同步一次云端计划，同步间隔为 24 小时。同步时间可安排在凌晨，并加入本地随机偏移，降低服务端请求集中度。

设备启动时也按一次唤醒周期处理。当前本地状态、同步时间、计划生效状态和显示内容共同决定本轮是否联网、是否刷新屏幕，以及下一次 deep sleep timer。

设备三个实体按键中预留一个作为手动调度键。按下该键后设备唤醒并进入同一条调度流程，本轮按额外强制同步处理：主动请求云端计划，重新判断计划状态和显示内容，并在流程结束后继续计算下一次 deep sleep timer。每日凌晨同步任务仍按原计划保留和计算，屏幕刷新仍由最终显示内容变化和异常保护决定。

屏幕刷新由计划日期和图片列表驱动。设备获取未来三天计划后，把下一次计划日期变化时间作为候选唤醒点。最终唤醒时间取下一次计划变化时间和下一次同步时间中更早的一个。

当天计划只有一张图片时，设备通常每天唤醒一次同步即可。计划内有多张图片时，设备按本地轮播间隔切换；资源已经缓存后，计划内切换不需要联网。

同步失败时，设备使用本地缓存计划继续运行。存在有效本地计划时，设备按计划显示，并安排短间隔重试同步；没有有效计划时，设备显示最后成功页面或整屏异常页，并采用逐步拉长的重试间隔。

## 缓存与空间

设备维护三类本地数据：

- 最近一次成功同步的计划文件。
- 图片资源索引。
- 最近一次显示状态。

TF 卡按 16GB 容量设计。设备不做日常缓存清理，当前显示项、未来三天计划引用资源和最后成功显示资源优先保留。

剩余空间低于 1GB 时触发空间保护清理。清理时删除未被当前计划和未来三天计划引用的历史图片，并按最近使用时间从旧到新处理，直到可用空间恢复到 3GB。

## 离线与低电量

离线时设备继续执行本地计划。计划过期后，设备保留最后成功显示的页面，并在下一次正常刷新时叠加计划过期通知；资源不可用时显示整屏异常页。

设备进入低电量状态后停止云端同步，继续使用本地缓存计划和已缓存图片。屏幕需要刷新时，设备在左上角叠加低电量通知，再进入休眠。电量恢复后，设备按正常同步流程重新获取云端计划。

## 通知与异常页

左上角通知用于表达轻量运行状态，只在屏幕本来需要刷新时写入。设备不为了普通状态变化单独刷新屏幕。右上角 `status` 区域为后续扩展保留，当前设备流程不写入右上角内容。

左上角通知包括：

- 低电量。
- 离线。
- 同步失败。
- 计划过期。
- 存储空间不足。

整屏异常页只处理无法继续显示照片的情况：

- 配置异常。
- TF 卡不可用。
- 无可用照片。

同步中、下载中、状态确认中等过程状态进入日志和运行状态记录。

## 时间处理

设备以本地 RTC 时间判断计划日期。成功同步计划时，可使用响应时间或 HTTP 头部时间修正本地日期。

计划日期使用 `YYYY-MM-DD` 字符串。当前项目按设备所在地日期处理，接口不引入时区字段。

长期离线导致时间不可信时，设备使用最后成功显示页面，并通过左上角通知表达计划或时间状态。

## 日志与状态

设备保存关键运行状态，便于串口排查：

- 最近一次成功同步时间。
- 最近一次同步失败原因。
- 当前计划 ID。
- 当前计划日期、图片 `sha256` 和图片位置。
- 当前标题。
- 下一次唤醒时间。
- 缓存资源数量和空间占用。
- 电量和充电状态。
- 最近一次屏幕刷新原因。
- 最近一次唤醒原因。

这些状态通过串口日志和本地状态文件查看。

## 风险与验证

屏幕刷新慢是主要体验约束。固件需要控制刷新次数，把联网、下载、状态确认和状态变化放在屏幕刷新之外处理。

纯 Rust 实现会增加底层驱动工作量。屏幕初始化、六色像素编码、图片解码、PMIC、电量读取和 TF 卡访问需要逐项验证，优先用最小功能样例打通硬件链路，再接入完整调度流程。

图片格式需要在目标硬件上验证解码耗时、颜色抖动和显示稳定性。云端优先输出 800 x 480 BMP，可降低设备端处理压力。

低功耗效果取决于 deep sleep、外设下电、PMIC 配置、Wi-Fi 连接时长和刷新次数。续航需要在目标硬件上实测。

## 结论

Epaper Album 的最终形态是一个个人使用的低功耗电子墨水相册。设备端用 Rust 实现计划同步、资源缓存、时间调度、照片渲染、通知合成和可靠休眠；云端负责未来三天计划和图片资源管理。复杂页面通过手工制作图片解决，设备端保持单一照片页面模型，固定合成左下角标题、右下角日期，并在需要时合成左上角通知。
