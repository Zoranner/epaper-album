# epaper-album Rust 工程评审报告

## 评审范围

本次评审只检查 Rust 工程体系、workspace 边界、共享协议契约、依赖和产物治理、质量门禁、文档与代码一致性。不评价业务需求完成度、硬件显示效果或单个功能 bug。

仓库当前是 ESP32-S3 固件、共享 `protocol` crate、Rust 服务端和 Vue 管理台组成的混合工程。根 `Cargo.toml` 同时定义固件 package 和 workspace，workspace members 包含 `crates/protocol` 与 `server`。

## 评审依据

- 用户提供的 `AGENTS.md`：Rust 修改后的验证要求、Node.js 使用 `bun`、文档正向表述和行尾治理要求。
- `review-engineering` 通用工程基线：目录边界、质量门禁、仓库治理、文档同步和报告落地要求。
- `review-rust` Rust 专项清单：Cargo workspace、crate 边界、features、错误处理、async、unsafe、build script、fmt、clippy、test、lockfile 和产物治理。
- 仓库事实：`Cargo.toml`、`Cargo.lock`、`crates/protocol/Cargo.toml`、`server/Cargo.toml`、`server/build.rs`、`.cargo/config.toml`、README、docs、Dockerfile、git 状态和命令输出。

## 总体结论

当前固件侧 Rust 格式、宿主目标 clippy 和固件侧测试基础较好，根目录 `cargo fmt --all -- --check`、`cargo clippy --all-targets --all-features -- -D warnings` 和根默认测试均已通过。

主要风险集中在工程边界和交付链路：公共接口文档仍保留旧的计划与 sprite 契约，服务端 Docker 构建上下文无法解析 workspace path dependency，根 workspace 默认命令容易漏掉 `server`，服务端测试又被本机正在运行的 `epaper-album-server.exe` 阻断。设备端 FFI `unsafe` 缺少安全前提说明，服务端生产配置默认值也需要收紧。

## 检查情况

- `git status --short`：执行前工作树干净。
- `cargo fmt --all -- --check`：通过。
- `cargo clippy --all-targets --all-features -- -D warnings`：通过，但 `cargo metadata` 显示根 workspace 默认成员只有根 package，因此该命令只覆盖默认根包，不能代表全 workspace。
- `cargo test --all-targets --all-features`：通过，109 个固件侧测试通过；同样只覆盖默认根包。
- `cargo clippy --workspace --all-targets --all-features -- -D warnings`：通过；服务端 `build.rs` 触发前端构建，输出 `[frontend] Building frontend...` 与 `[frontend] Done.`。
- `cargo test --workspace --all-targets --all-features`：未通过验证。命令在 Windows 上删除 `target\debug\epaper-album-server.exe` 时失败，错误为 `拒绝访问。 (os error 5)`。
- `cargo test -p epaper-album-server --lib --tests`：未通过验证，失败点同样是删除 `target\debug\epaper-album-server.exe` 被拒绝。
- `Get-Process` 检查到 `epaper-album-server.exe` 正在运行，路径为 `D:\hardware\epaper-album\target\debug\epaper-album-server.exe`，该进程占用解释了服务端测试无法覆盖的直接原因。
- `git ls-files` 与 ignore 规则检查：`target/`、`.embuild/`、`.refs/`、根 `data/`、`server/data/`、`server/.env`、`server/web/dist/`、`server/web/node_modules/` 和 `server/assets/fonts/*` 均被忽略；当前跟踪文件未发现明显构建产物、真实密钥或大字体文件。
- `.gitattributes`、`rust-toolchain.toml`、`rustfmt.toml`、`clippy.toml`、`.editorconfig` 均不存在。

## 主要问题

### 公共计划接口文档与代码契约漂移

问题定性：违反硬性工程基线。

`docs/epaper-album-design.md` 与 `docs/epaper-album-server-design.md` 中仍描述 `id`、`start`、`end`、`images` 等旧的多图计划结构；当前共享协议已经是单图计划结构。`crates/protocol/src/lib.rs` 中的 `Plan` 使用 `date`、`caption`、`image` 字段，服务端数据库行也是 `date`、`caption`、`image`，前端 API 类型同样使用这组字段。

风险是设备端、服务端、前端或外部调试工具按文档接入时会发送旧字段，导致联调失败或形成新的兼容分支。计划接口是公共契约，应以 `protocol` crate 的类型为准，同步修订设备设计、服务端设计、Bruno 示例和 README 中的接口说明。

### sprite 接口文档与实现不一致

问题定性：违反硬性工程基线。

文档仍描述 `GET /api/sprite?...` 直接返回 BMP。当前实现是设备先请求 `/api/sprites` 获取 sprite 元数据，再按 `/sprites/{sha256}` 下载 BMP；服务端路由也只注册了 `/api/sprites` 和 `/sprites/:sha256`。

风险是 sprite 生成、缓存和下载链路的真实契约被文档误导。建议把文档统一改为“元数据接口 + 资源下载接口”，并明确 caption、date、status 三类 sprite 的 sha256 缓存策略。

### 服务端 Docker 构建上下文无法覆盖 workspace path dependency

问题定性：违反硬性工程基线。

README 要求进入 `server/` 后执行 `./docker-build.sh`。该脚本以 `server/` 为 Docker build context，并使用 `server/Dockerfile`。但 `server/Cargo.toml` 依赖 `protocol = { path = "../crates/protocol" }`，Dockerfile 只复制了 `server/Cargo.toml`、`server/Cargo.lock`、`server/build.rs` 和 `server/src/`，没有复制 workspace 根与 `../crates/protocol`。

风险是容器发布链路无法复现，镜像构建阶段无法解析 path dependency。建议把 Docker build context 提升到仓库根，或调整 Dockerfile 复制 workspace 根 `Cargo.toml`、根 `Cargo.lock`、`crates/protocol` 和 `server`，并以 workspace 成员方式构建服务端。

### 根 workspace 默认质量门禁容易漏掉服务端

问题定性：规范缺失导致风险。

根 `Cargo.toml` 同时定义 package 和 workspace，members 包含 `crates/protocol` 与 `server`，但未声明 `default-members`。`cargo metadata` 显示 `workspace_default_members` 只有根 `epaper-album`。因此在根目录执行 `cargo clippy --all-targets --all-features` 或 `cargo test --all-targets --all-features` 只覆盖固件根包，不覆盖 `server` 和 `protocol` 的完整 workspace 视角。

风险是日常验证容易把“默认根包通过”误判成“全仓通过”。建议在文档中把全仓命令写成 `cargo clippy --workspace --all-targets --all-features -- -D warnings` 与 `cargo test --workspace --all-targets --all-features`，并考虑在根 `Cargo.toml` 中显式声明 `default-members`。

### 设备端 FFI unsafe 缺少安全前提说明

问题定性：违反硬性工程基线。

`src/platform/wifi.rs` 中直接通过 `unsafe` 调用 `esp_wifi_set_ps(0)`；`src/power/mod.rs` 中多处通过 `unsafe` 调用 GPIO 配置、GPIO 电平读取、重启和 wakeup status 读取。嵌入式 FFI 调用可以存在，但当前代码没有在封装点说明调用前提、硬件状态要求、线程或生命周期边界。

风险是后续调整启动顺序、电源管理或按键逻辑时，维护者难以判断这些 unsafe 调用是否仍满足 ESP-IDF API 前提。建议把底层 unsafe 封装集中到平台适配层，并在每个封装点用简短注释写清安全前提。

### 服务端默认凭据与默认监听范围组合风险较高

问题定性：规范缺失导致风险。

`server/src/config.rs` 在缺少环境变量时使用 `local-secret-key`、`admin`、`admin`，同时默认监听 `0.0.0.0:3000`。README 已说明正式部署需要编辑 `server/.env`，但代码层没有区分开发和部署环境。

风险是部署时遗漏 `.env` 会让服务以默认管理员账号和默认密钥对外监听。建议增加显式运行环境变量，生产环境缺少 `SECRET_KEY`、`ADMIN_USERNAME`、`ADMIN_PASSWORD` 时直接启动失败；开发环境可以继续保留本地默认值，并默认绑定 `127.0.0.1`。

### 服务端数据库日期字段缺少持久化约束

问题定性：建议优化。

`server/src/db.rs` 创建 `plans.date` 为 `TEXT PRIMARY KEY`，没有日期格式 CHECK 约束；读取时 `plan_from_row` 使用 `LocalDate::parse(&row.date).expect("stored plan date must be valid")`。接口层能校验日期格式，但持久层不能防御手工修库、旧数据迁移或异常导入。

风险是列表查询等读取路径可能因一条脏数据 panic。建议为 `plans.date` 增加格式约束或在读取路径返回可控错误，并补一条脏数据测试。

### 服务端 Cargo build script 扩大 Rust 验证副作用

问题定性：规范缺失导致风险。

`server/build.rs` 在 `web/node_modules` 不存在时执行 `bun install`，随后执行 `bun run build`。这使 `cargo build`、`cargo clippy`、`cargo test` 都可能受 Bun、前端依赖、网络和前端构建状态影响。仓库提供了 `SKIP_FRONTEND_BUILD=1`，但 README 中的常用验证命令没有把“后端静态检查/测试”和“完整打包构建”明确分开。

风险是 Rust 后端验证入口不够稳定，CI 或本地审查容易被前端环境影响。建议文档明确两组命令：后端快速验证使用 `SKIP_FRONTEND_BUILD=1`，完整发布构建再执行前端构建和服务端 release 构建。

### 行尾和文本属性缺少仓库级治理

问题定性：规范缺失导致风险。

仓库不存在 `.gitattributes` 和 `.editorconfig`。子任务检查 `git ls-files --eol` 时发现部分已跟踪文件存在 mixed 行尾状态。当前没有未提交 diff，因此这不是本次改动造成的问题，也不应在本次评审里直接规范化。

风险是 Windows 与 Linux 协作、Docker 构建脚本、Shell 脚本和 Rust/TS 文件容易产生无关格式噪音。建议后续单独做格式治理任务，先定义 `.gitattributes` 和 `.editorconfig`，再独立提交行尾规范化。

### 前端管理台缺少独立 lint、格式检查和测试入口

问题定性：建议优化。

`server/web/package.json` 只有 `dev`、`build`、`preview` 脚本。`build` 包含 `vue-tsc --noEmit && vite build`，能覆盖类型检查和构建，但没有独立 lint、format check 或组件测试入口。

风险是管理台改动只能靠构建发现类型和打包问题，代码风格、复杂交互回归和组件边界缺少自动化反馈。建议补充前端 lint、format check 和必要的组件/交互测试入口，并继续使用 `bun`。

## 整改优先级建议

### 第一阶段：修正交付链路和公共契约

- 统一计划接口文档到 `date`、`caption`、`image` 单图结构。
- 统一 sprite 文档到 `/api/sprites` 元数据接口和 `/sprites/{sha256}` 下载接口。
- 修复服务端 Docker 构建上下文，使镜像构建能解析 `crates/protocol` path dependency。
- 把 README 与设计文档中的全仓 Rust 验证命令改为显式 `--workspace`。

### 第二阶段：补齐验证边界和运行安全

- 处理或关闭当前占用 `target\debug\epaper-album-server.exe` 的本机服务端进程后，重跑 `cargo test --workspace --all-targets --all-features`。
- 对服务端生产配置增加强校验，避免默认凭据绑定全部网卡。
- 将服务端 Rust 快速验证和完整前端打包构建拆成清晰命令。
- 为 `plans.date` 增加持久化约束或可控错误路径。

### 第三阶段：治理长期维护风险

- 为 ESP-IDF FFI unsafe 封装补充安全前提说明。
- 单独制定 `.gitattributes`、`.editorconfig` 和行尾规范化提交。
- 为前端管理台补充 lint、format check 和必要测试入口。
