# epaper-album 结构与代码体量专项评审报告

## 评审范围

本次评审是对 `2026-06-19-Rust工程评审.md` 的结构与代码体量专项补充。评审按两个项目分开：

- 设备端项目：`device/` 固件工程、`device/src/`、`crates/protocol`、`device/scripts/`、设备端设计文档和嵌入式构建配置。
- 服务端项目：`server/` Rust 后端、`server/web/` Vue 管理台、`server/docs/`、Docker 与服务端构建脚本。

`crates/protocol` 是设备端和服务端共同依赖的协议契约，不应被服务端实现或设备端实现单方面吞并。本次检查把 Rust、Vue、TypeScript、CSS、PowerShell、Shell、Dockerfile、`build.rs`、Bruno 示例和工程文档都作为代码资产或工程资产纳入评审；脚本也是代码。

本次评审基于当前工作树快照。评审时工作树已有未提交改动，且评审期间仍有并行改动进入工作树；这些业务代码改动不是本次评审产生的，本报告只做只读扫描和报告落地。后续整改应先以当时的 `git status` 和 `git diff` 重新确认实际基线。

## 评审依据

- 用户要求：设备端和服务端是两个工程、两个项目，评审和整改不能混在一起；脚本必须作为代码资产评审。
- `review-engineering` 通用工程基线：目录边界、入口层职责、大文件多职责、验证入口、仓库治理和报告落地。
- `review-rust` Rust 专项基线：Cargo 边界、crate 职责、测试组织、构建脚本、fmt、clippy 和质量门禁。
- 仓库事实：设备端当前位于 `device/`，`device/Cargo.toml` 是独立固件工程，并通过 path dependency 引用 `../crates/protocol`；`server/Cargo.toml` 是独立服务端工程，并通过 path dependency 引用 `../crates/protocol`。
- 当前文件统计：使用 Git 跟踪文件过滤源码、脚本、配置和文档后统计行数；本地 `node_modules`、`dist`、`data`、`target`、`.embuild` 和 `.refs` 已按忽略规则排除，不作为项目源码体量统计。

## 总体结论

当前仓库已经在 Cargo 层面把设备端和服务端拆成两个独立 Rust 工程，但目录和代码组织还没有完全跟上这个边界。设备端项目的问题集中在核心业务周期、渲染链路、平台适配和电源策略文件过大；服务端项目的问题更集中，后端 `routes/mod.rs`、`db.rs` 和 `tests/core.rs` 把多个领域塞进单文件，目录结构没有承载真实模块边界。

服务端结构风险高于设备端：`server/src/routes/` 目录只有一个 1271 行的 `mod.rs`，同时承担路由注册、鉴权、计划、图片、sprite、后台处理、路径工具、图片处理和校验映射；`server/tests/core.rs` 达到 1294 行，所有集成测试和测试夹具混在一起。设备端虽然已有 `app`、`graphics`、`platform`、`power`、`sync` 等目录，但单个模块仍承担多个层级职责，后续维护会继续受到大文件影响。

前端管理台的 Vue 组件体量暂未形成首要风险，但 `server/web/src/styles/components.css` 已达到 771 行，说明样式系统也需要作为代码资产治理。脚本文件当前没有超过 450 行，`device/scripts/prepare-test-bmp.ps1` 为 156 行、`device/scripts/activate-esp-idf.ps1` 为 56 行；脚本暂无体量风险，但仍应纳入代码评审和后续质量门禁。

## 设备端项目主要问题

### 渲染模块体量过大且职责混杂

问题定性：违反硬性工程基线。

`device/src/graphics/render.rs` 当前 1306 行，文件内同时包含文本样式、overlay slot、渲染错误、内置错误页输入、sprite BMP、packed frame 渲染、照片页渲染、文字换行、glyph 绘制、BMP 贴图、sprite 贴图、坐标计算和大量单元测试。

风险是渲染管线、错误页布局、字体像素工具和 sprite 合成互相牵连。后续只修改错误页或 sprite 叠加时，也必须理解整条渲染链路，回归测试范围变大。

建议按职责拆分为 `graphics/render/mod.rs` 组合入口，并拆出 `photo.rs`、`error_page.rs`、`text.rs`、`glyph.rs`、`sprite.rs`、`packed_frame.rs` 和对应测试模块。公开 API 先保持在 `render/mod.rs` re-export，降低调用方迁移成本。

### 设备运行周期模块承担应用层总控和决策细节

问题定性：违反硬性工程基线。

`device/src/app/cycle.rs` 当前 1054 行，文件内同时定义设备周期输入、云同步 trait、显示 trait、同步决策、显示决策、运行上下文、主运行函数、刷新请求、结果归类和大量测试。

风险是应用层入口既负责流程编排，又负责同步/显示决策细节。设备端后续调整低电量策略、缓存显示策略或同步错误处理时，容易在一个文件里叠加条件分支。

建议拆出 `app/sync_decision.rs`、`app/display_decision.rs`、`app/outcome.rs`、`app/runner.rs` 和 `app/ports.rs`。`run_device_cycle` 保留为组合入口，纯决策函数独立测试。

### ESP-IDF 平台适配层聚合过多硬件能力

问题定性：建议优化。

`device/src/platform/espidf.rs` 当前 742 行，聚合了设备周期启动、存储挂载、诊断日志、配置读取、云同步适配、Wi-Fi 错误映射、周期文件写入、时间读取和下次运行计划构造。

风险是平台适配层从“适配硬件能力”扩张成“设备运行总装配层”，硬件能力、文件系统、网络同步和诊断报告交织在一起。后续替换显示屏、调整存储路径或扩展诊断日志时，影响面不清晰。

建议按 capability 拆成 `platform/espidf/runtime.rs`、`storage_mount.rs`、`diagnostics.rs`、`cloud_sync.rs`、`config_io.rs` 和 `time.rs`。平台层保留硬件调用和装配，业务决策继续留在 `app`、`sync`、`power` 等模块。

### 电源模块同时包含策略逻辑和 ESP-IDF 控制细节

问题定性：建议优化。

`device/src/power/mod.rs` 当前 618 行，既包含 `BatteryStatus`、`PowerProfile`、`LowBatteryPolicy`、下次运行时间计算等纯策略，又在同文件内包含 `espidf` 子模块，处理 GPIO、deep sleep、restart、wakeup status 等平台细节。

风险是可单元测试的电源策略和只能在硬件目标上验证的平台控制混在一起，后续做功耗策略变更时容易误触平台调用。

建议将纯策略保留为 `power/policy.rs`、`power/schedule.rs`，ESP-IDF 实现迁移到 `power/espidf.rs` 或 `platform/espidf/power.rs`。策略模块只暴露可测试的纯函数和数据结构。

### 设备端同步模块仍可继续拆分

问题定性：建议优化。

`device/src/sync/device.rs` 当前 579 行，包含 `CloudResourceSync`、同步错误、云计划同步、图片资源检查、sprite 同步和测试。`device/src/sync/cloud.rs`、`device/src/sync/resource.rs` 也分别达到 427 行和 365 行，说明同步域已经有拆分基础，但设备侧编排仍偏集中。

风险是计划同步、资源下载、缓存可用性检查和 sprite 资源策略互相影响。后续资源类型增加时，`device.rs` 会继续膨胀。

建议将 `device.rs` 拆为 `sync/device/mod.rs`、`plans.rs`、`resources.rs`、`sprites.rs`、`errors.rs` 和测试模块。同步错误映射和缓存可用性判断应分别归位，避免继续堆在设备同步入口。

### 设备端文档体量已接近代码治理对象

问题定性：建议优化。

`device/docs/device-runtime-policy-design.md` 当前 534 行。它是设备端运行策略的重要工程资产，已经超过多个源码文件体量。

风险是运行策略、功耗策略、异常策略、缓存策略和实现约束都堆在单一长文档中，后续文档更新容易遗漏和代码变更的对应关系。

建议后续按“设备运行周期”“缓存与同步”“功耗与唤醒”“异常与提示”拆分文档，或至少建立目录索引和代码锚点。文档拆分应和代码拆分同步推进，避免评审、实现和文档三套边界不一致。

## 服务端项目主要问题

### 后端路由目录没有承载真实模块边界

问题定性：违反硬性工程基线。

`server/src/routes/mod.rs` 当前 1271 行，`server/src/routes/` 目录下只有这一个文件。该文件同时包含：

- `AppState`、`AdminSession`、运行时状态和处理队列。
- 路由注册、健康检查、登录、计划接口、图片接口、sprite 接口和下载接口。
- 后台图片处理队列、图片格式探测、缩放、六色量化和 BMP 编码。
- sprite 字体配置、字体加载、sprite 渲染、sprite 缓存和 sha256 计算。
- 文件路径工具、权限校验、参数解析、tag 归一化、日期校验、sha256 校验和错误映射。
- 模块内测试。

风险是服务端后端所有核心领域都被压在一个模块里，目录结构虽然有 `routes/`，但没有形成 `auth`、`plans`、`images`、`sprites`、`processing`、`files` 等真实边界。新增任一接口都会继续放大这个文件。

建议第一优先级拆分 `server/src/routes/`：`mod.rs` 只负责组合 router；`auth.rs` 放登录和权限；`plans.rs` 放计划接口；`images.rs` 放图片上传、列表、下载和删除；`sprites.rs` 放 sprite 元数据和下载；`processing.rs` 放处理队列与图片处理；`paths.rs` 放路径工具；`validation.rs` 放参数和 payload 校验。

### 服务端集成测试集中在单文件

问题定性：违反硬性工程基线。

`server/tests/core.rs` 当前 1294 行，包含测试应用启动、请求 helper、登录 helper、图片 seed、计划 seed、multipart 构造、sprite 字体资产探测，以及鉴权、计划、图片、上传、下载、sprite、schema 迁移等所有集成测试。

风险是测试文件已经成为第二个应用入口。测试夹具和领域测试混杂后，单个测试失败时定位成本高，也不利于后续按功能模块并行维护。

建议拆成 `server/tests/common/mod.rs`、`auth.rs`、`plans.rs`、`images.rs`、`sprites.rs`、`schema.rs`、`processing.rs`。公共 helper 放 `common`，各测试文件只保留本领域断言。

### 数据访问层单文件承担 schema、迁移和领域查询

问题定性：建议优化。

`server/src/db.rs` 当前 610 行，包含 `Store`、计划查询、图片查询、tag 处理、schema 初始化、旧表替换、列检查、行映射和数据规范化。

风险是数据库结构演进、迁移兼容和业务查询耦合在一个文件里。后续继续增加图片处理状态、计划筛选或资源索引时，`db.rs` 会同时承担 schema owner 和 repository owner。

建议拆成 `db/mod.rs`、`schema.rs`、`migrations.rs`、`plans.rs`、`images.rs`、`tags.rs` 和 `mapping.rs`。`Store` 可以作为门面保留，具体 SQL 按领域分文件。

### 前端样式文件超过代码体量阈值

问题定性：建议优化。

`server/web/src/styles/components.css` 当前 771 行，是服务端项目内除 Rust 后端大文件外最大的前端代码资产。Vue 视图文件 `ImagesView.vue` 265 行、`PlansView.vue` 226 行，暂未形成同等级体量风险。

风险是组件样式逐步成为全局样式仓库，具体业务组件的视觉规则、通用控件规则和页面布局规则容易混在一起。后续改一个组件样式可能影响多个页面。

建议按现有前端目录拆分样式：`styles/components/base.css`、`input.css`、`overlay.css`、`images.css`、`plans.css`、`navigation.css`。全局 token 和布局继续留在 `base.css`、`layout.css`，业务组件样式靠近对应组件目录或按组件域拆分。

### 服务端文档和接口示例也需要按项目治理

问题定性：建议优化。

`server/docs/epaper-album-server-design.md` 当前 708 行，内容属于服务端项目设计。`server/docs/` 下已有管理台前端设计和 Bruno 示例，说明服务端文档已经按项目边界收敛到服务端目录。

风险是服务端文档散落在根 `docs/` 和 `server/docs/` 两处，和“两个项目”边界不一致。后续服务端 API 变更时，维护者很难判断应更新根文档还是服务端文档。

建议把服务端专属文档逐步归入 `server/docs/`，根 `docs/` 保留产品级、设备端和跨项目协议文档。跨项目契约建议单独放 `docs/protocol.md` 或 `crates/protocol/README.md`，明确它不是服务端私有实现。

### 服务端脚本体量不大但仍应纳入质量门禁

问题定性：建议优化。

服务端 `server/build.rs`、`server/docker-build.sh`、`server/Dockerfile` 和 compose 文件行数不大，但这些文件决定构建、打包和部署链路，属于代码资产。

风险是只评审 Rust handler 和数据库代码时，容易漏掉 build script、Docker context、前端构建副作用和包管理器约束。原 Rust 评审已经指出 Docker 构建上下文和 `build.rs` 副作用问题，本次结构复评确认这些文件应纳入后续整改计划。

建议后续服务端整改清单中固定包含脚本和构建资产检查项：Docker build context、`build.rs` 前端构建触发条件、`bun` 命令入口、`.dockerignore`、compose 示例和 Bruno 示例。

## 跨项目边界问题

### 两个项目边界需要在目录和文档中显式表达

问题定性：规范缺失导致风险。

当前目录结构已经体现边界：设备端工程位于 `device/`，服务端工程位于 `server/`，共享协议位于 `crates/protocol`。文档、评审和整改计划仍需要持续按“设备端 + 服务端 + 共享协议”分别归口，避免把三者当成一个 Rust 工程整体处理。

风险是后续整改时把设备端验证、服务端验证、前端验证和协议契约验证混在一起，既影响验证速度，也会让提交粒度失控。

建议建立两个项目的固定边界说明：

- 设备端项目：`device/Cargo.toml`、`device/src/`、`crates/protocol` 的协议消费、`device/scripts/` 和设备端文档。
- 服务端项目：`server/Cargo.toml`、`server/src/`、`server/web/`、`server/docs/`、服务端 Docker 和部署脚本。
- 共享协议：`crates/protocol` 作为跨项目契约，由两端共同验证，不塞入任一项目的业务实现目录。

### 代码体量阈值应覆盖所有代码资产

问题定性：规范缺失导致风险。

本次扫描确认超过 450 行的不只 Rust 文件，还包括 CSS 和工程文档。脚本当前未超过阈值，但脚本、构建脚本、Dockerfile、Bruno 示例、CSS、Vue、TypeScript 都应按代码资产看待。

风险是后续评审如果只看 Rust 源码，会继续漏掉前端样式、构建脚本、接口示例和部署脚本中的工程问题。

## 整改优先级建议

### 第一阶段：先拆服务端后端结构

- 拆分 `server/src/routes/mod.rs`，让 `mod.rs` 只组合路由。
- 拆分 `server/tests/core.rs`，把测试夹具和各领域集成测试分离。
- 拆分 `server/src/db.rs` 的 schema、迁移、计划、图片和 tag 查询。
- 每次拆分保持接口行为不变，按服务端项目单独运行 Rust 和前端验证，不夹带设备端改动。

### 第二阶段：拆设备端应用决策和同步边界

- 拆分 `device/src/app/cycle.rs` 的端口定义、同步决策、显示决策、运行结果和 runner。
- 拆分 `device/src/sync/device.rs` 的计划同步、资源同步、sprite 同步和错误映射。
- 保持纯决策函数先行，硬件平台适配后置，避免一次性改动设备运行链路。

### 第三阶段：治理渲染、平台和样式大文件

- 拆分 `device/src/graphics/render.rs` 的照片页、错误页、文本/glyph、sprite 和 packed frame 渲染。
- 拆分 `device/src/platform/espidf.rs` 与 `device/src/power/mod.rs` 中的平台能力和纯策略逻辑。
- 拆分 `server/web/src/styles/components.css`，把通用组件样式和业务域样式分开。
- 整理服务端专属文档到 `server/docs/`，根 `docs/` 保留设备端、产品级和跨项目契约文档。

### 第四阶段：建立持续检查规则

- 新增或补充代码体量扫描脚本，把 Rust、Vue、TypeScript、CSS、PowerShell、Shell、Dockerfile、`build.rs`、Bruno 示例和工程文档都纳入统计。
- 对超过 450 行的文件要求在评审报告中给出职责说明；超过 800 行要求列入拆分计划；超过 1000 行的业务代码、路由和测试文件进入优先整改。
- 按两个项目分别维护验证入口：设备端验证、服务端后端验证、服务端前端验证和共享协议验证分别列出，不再用一个“Rust 工程通过”覆盖全部。
